# `sz_presence_wake` 具体脚本蓝图

这份文档是在 [01_sz_presence_wake_fixed.md](./01_sz_presence_wake_fixed.md) 的基础上，再往前推进一层。

它专门回答：

## 参考依据

### 视频

- `docs/Shenzhen/Videos/01_20260425_043820_01.mp4`

这段视频给出的不是最终代码真值，而是：

- 节奏参考
- 情绪张力参考
- “不是一步到位机械抬起”的演出感觉参考

从抽帧看，这一段最重要的特征是：

1. 从静止到注意力启动是缓慢的
2. 中间存在明显的“中间位”
3. 有一个高位停顿
4. 最后不是直接停死，而是进入“看向人”的状态

### PDF 第一条

- `docs/Shenzhen/Videos/Mira Light 展位交互方案 副本.pdf`

第一页到第二页讲的是：

1. 静止状态
2. 灯先缓缓亮起微光
3. 灯臂缓缓升起，灯头慢慢抬起来
4. 到最高处做一次“伸懒腰”
5. 再慢慢回到正常高度
6. 回到正常高度时抖两下
7. 最后慢慢转头看向评委

这说明这个场景的核心不在“复杂”，而在：

- 分阶段
- 有停顿
- 有一点生命感

## 当前版本的总体策略

当前阶段不要做实时 presence wake。

第一版目标是：

- 固定起始睡姿
- 固定灯光唤醒
- 固定分三段抬起
- 固定双抖动
- 固定朝一个方向看过去

只要这套固定版能稳定演出来，就已经足够用于：

- 主秀排练
- 主持人口播配合
- 动作节奏校准

## 推荐脚本名

如果落到板端或未来 `shenzhen/fixed` 对应的脚本文件，建议名字直接叫：

```text
sz_presence_wake_fixed.py
```

如果先只写在 repo 层，建议未来对应：

```text
scripts/scenes_shenzhen.py -> sz_presence_wake
```

## 推荐动作分段

建议拆成 **6 段**。

### 第 0 段：起始睡姿

目标：

- 所有演示从同一个睡姿起点开始

要求：

- 不在这里做任何“聪明感”计算
- 只是确保每次起床都从一致状态出发

推荐来源：

- `sleep_motion.py`
- 或固定 pose `sleep_pose`

如果第一版需要更稳，建议直接约定：

- 每轮演示前人工保证灯已在 sleep 附近

## 第 1 段：微光亮起

目标：

- 先让人感知“睁眼了”
- 机械臂暂时不动

推荐命令：

```bash
python3 /home/sunrise/Desktop/send_uart3_led_cmd.py wake 255 220 180 150
```

如果觉得 `WAKE` 太完整，也可以改成两步式低亮暖光：

```bash
python3 /home/sunrise/Desktop/send_uart3_led_cmd.py all 255 220 180 40
sleep 0.4
python3 /home/sunrise/Desktop/send_uart3_led_cmd.py all 255 220 180 90
```

但从维护角度，第一版优先建议直接复用 `WAKE`。

建议时长：

- `0.4s ~ 0.8s`

## 第 2 段：半醒抬头

目标：

- 不直接冲到最高位
- 先到一个“半醒姿态”

这一段的作用是：

- 给“醒来”留中间层次
- 让后面的高位伸展更有戏

第一版可以先做成一个固定姿态，例如：

```text
servo0 -> 保持当前朝向或近中位
servo1 -> 抬起到中间位
servo2 -> 前段适度抬起
servo3 -> 灯头略微抬起
```

板端种子建议参考：

- `servo_12_slow_to_1800_2750.py`
- `four_servo_pose_delay_2.py`

建议时长：

- `0.8s ~ 1.2s`

## 第 3 段：高位伸懒腰

目标：

- 到达高位
- 灯头稍微后仰
- 停一下

这是这一幕里最重要的“情绪高点”。

不要一笔带过。

推荐实现方式：

- 直接定义一个固定 `wake_high_pose`
- 到位后停 `0.8s ~ 1.2s`

板端种子可参考：

- `four_servo_pose_2048_2048_2048_2780_separate.py`
- `four_servo_pose_delay_2.py`

建议时长：

- 上升：`0.8s ~ 1.2s`
- 顶部停顿：`1.0s`

## 第 4 段：回到正常高度

目标：

- 从高位回到工作位
- 给后面的“抖两下”腾出空间

这一段不要做成骤降。

必须是：

- 稳定落回
- 再进入微抖

建议时长：

- `0.6s ~ 1.0s`

## 第 5 段：抖两下

目标：

- 表达“刚醒来的小动物抖抖毛”
- 幅度小
- 节奏清晰

第一版不要直接用大幅摇头脚本粗暴替代。

更推荐：

- 基于 `servo_2_nod_1900_2200.py`
- 基于 `servo_3_shake_2100_2000.py`

只取它们的“小幅、慢速、两拍”思路，在脚本里做更轻的双拍。

建议形态：

- 一次微小下压 / 回弹
- 再一次微小左右微摆 / 回正

建议总时长：

- `0.5s ~ 0.8s`

## 第 6 段：看向评委

目标：

- 用一个明确但缓慢的朝向收尾
- 让“醒来”变成“注意到你”

当前阶段不要接实时 `targetDirection`。

先固定一个默认方向：

- 例如主持人通常站位
- 或默认右前方

推荐实现方式：

- 用一个固定 look pose 收尾
- 不要再加多余小动作

建议时长：

- `0.5s ~ 0.8s`

## 第一版最推荐的落地方式

不要一上来就把所有低层动作都重新发明一遍。

更稳的做法是：

1. 先把现有板端脚本拆成“可复用阶段”
2. 再用一个上层 fixed script 把它们顺序组合

也就是说，第一版不是：

- 重新发明控制协议

而是：

- 利用现有板端动作种子
- 编排成 PDF 第一条要求的舞台节奏

## 当前固定版能证明什么

它能证明：

- Mira 会醒来
- Mira 醒来的过程有层次
- Mira 的灯与身体是联动表达
- Mira 最后会把注意力放到人身上

它还不能证明：

- Mira 是因为真的检测到来人才醒来
- Mira 会按实时来人方向选择最终朝向

## 第二版再做什么

等第一版固定动作稳定后，再升级：

1. 最后一段的 `look_to_guest()` 改成 `targetDirection`
2. 再把整个 scene 的触发条件接到 presence event

## 一句话总结

这条“起床”场景第一版最正确的做法不是追求实时，而是：

> 用固定灯效、固定三段抬起、固定一秒停顿、固定双抖和固定收尾朝向，把 PDF 第一条先稳定演出来。
