# 展位图像摘要记忆观察器

这篇说明对应仓库里的新脚本：

- [scripts/capture_memory_observer.py](/Users/huhulitong/Documents/GitHub/Mira-Light/scripts/capture_memory_observer.py)
- [scripts/run_capture_memory_observer.sh](/Users/huhulitong/Documents/GitHub/Mira-Light/scripts/run_capture_memory_observer.sh)

它的目标不是高频 tracking，而是每 `5` 分钟从 `captures/` 里挑一张更适合沉淀记忆的图片，做一次结构化视觉总结，然后把结果写进当前云端 `Claw / Mira_v3` 记忆体系。

## 作用边界

这条链路负责：

- 从 `capture` 落盘目录里挑一张相对清晰、稳定、信息量更高的 JPG
- 调火山多模态模型总结这张图中的人、物、时间、地点
- 通过 `memory-context` 写入 `episodic + working + session-memory`
- 可选地把同一份 JSON 通过 SSH 丢到云端 `mira-openclaw` 目录，给云端流程继续消费

这条链路不负责：

- 替代实时 CV tracking
- 对每一帧都做大模型分析
- 直接控制舵机

## 推荐链路

```text
capture JPGs
-> capture_memory_observer.py
-> pick one representative image
-> Volc multimodal summary
-> memory-context write
-> cloud prompt-pack / session-memory retrieval
-> Mira / cloud Claw sees the observation as context
```

如果已经有云端 `memory-context`，优先走这条路。

原因很简单：

- 这和当前云端 `Claw` 记忆系统边界一致
- `memory-context + session-memory + prompt-pack` 已经是仓库确认过的主接入面
- 不需要要求云端一定暴露某个固定 `openclaw` CLI

## 当前写入策略

脚本每次会做两层写入：

- `session-memory`
  保存“最近一次代表性图像观察”的当前状态
- `memory-context`
  只在观察结果足够有价值时写入

写入的记忆层如下：

- `episodic / environment_observation`
- `working / scene_state`

“足够有价值”当前的判断标准是：

- 模型显式标记 `memoryWorthy=true`
- 或图中出现了人
- 或图中有多个主要物体
- 且没有在短时间内和上一条记忆高度重复

## 模型输出要求

脚本要求火山多模态模型返回严格 JSON，核心字段是：

- `peopleCount`
- `peopleSummary`
- `objects`
- `activity`
- `sceneSummary`
- `time`
- `location`
- `memoryWorthy`
- `memoryReason`

其中：

- `time` 用图片文件时间
- `location` 用公网 IP 反查结果
- 人和物都要求保守总结，不允许编造

- `MIRA_LIGHT_VOLC_API_KEY`
- `MIRA_LIGHT_VOLC_MODEL`
- `MIRA_LIGHT_MEMORY_CONTEXT_BASE_URL`

- `MIRA_LIGHT_CLOUD_CLAW_SSH_TARGET`
- `MIRA_LIGHT_CLOUD_CLAW_DROP_DIR`

- `doubao-seed-2-0-lite-260215`

默认 capture 目录是：

- `~/.openclaw/workspace/runtime/captures`

默认本地运行目录是：

- `~/.openclaw/workspace/runtime/capture-memory-observer`

## 直接运行

先准备：

```bash
export MIRA_LIGHT_VOLC_API_KEY='...'
export MIRA_LIGHT_MEMORY_CONTEXT_BASE_URL='https://<your-memory-context-host>'
export MIRA_MEMORY_CONTEXT_AUTH_TOKEN='...'
export MIRA_LIGHT_MEMORY_CONTEXT_USER_ID='mira-light-bridge'
```

做一次单轮验证：

```bash
bash scripts/run_capture_memory_observer.sh --once --log-level INFO
```

如果你只想看本地挑图和产物，不想真的调用模型或写记忆：

```bash
bash scripts/run_capture_memory_observer.sh --once --dry-run --log-level DEBUG
```

## 云端 Claw 接法

按当前仓库边界，最推荐的顺序是：

1. 本地脚本先把图像观察写进云端 `memory-context`
2. 云端 `Claw / Lingzhu` 再通过 `prompt-pack` 和 `session-memory` 看见这条观察
3. 只有在你确实还需要一份云端侧文件落地时，再打开 SSH handoff

对应可选配置：

```bash
export MIRA_LIGHT_CLOUD_CLAW_SSH_TARGET='ubuntu@43.160.239.180'
export MIRA_LIGHT_CLOUD_CLAW_DROP_DIR='/home/ubuntu/mira_import/mira/.mira-runtime/mira-openclaw/var/capture-observations'
```

这一步不会替代 `memory-context`，它只是额外给云端流程一份可消费 JSON。

## 本地产物

脚本会在运行目录下写两类文件：

- `prompts/*.json`
  发给模型的请求
- `observations/*.json`
  标准化后的观察结果

同时会维护：

- `state.json`

它用于避免每一轮都重复写同一类记忆。

## 现场建议

- 先用 `--dry-run` 确认挑图逻辑和落盘目录都正常
- 再打开真实模型调用，检查 JSON 是否稳定
- 最后再接云端 `memory-context`

如果现场网络不稳，优先保证：

- 图片筛选脚本继续运行
- 本地 artifact 继续落盘

这样即使临时写不进云端，后面也还能回放或补写。
