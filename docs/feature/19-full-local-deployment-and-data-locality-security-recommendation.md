# 完整本地部署、数据不出本地与高安全边界建议

## Current Status

与“云端认知、本地身体”的方案相对，`Mira Light` 在合适条件下也可以采用另一条明确路线：

```text
完整本地部署
-> 数据不出本地
-> 本地闭环执行
-> 最强本地安全边界
```

这条路线的核心不是“云端更聪明”，而是：

- 推理留在本地
- 记忆留在本地
- bridge 留在本地
- runtime 留在本地
- 语音、视觉与设备执行都留在本地

因此，它更适合那些把隐私、合规、内网可控性和物理现场稳定性放在首位的场景。

## What This Document Argues

这篇文档要表达的核心判断是：

- 在合适条件下，`Mira Light` 可以实现完整本地部署
- 这样可以实现“数据不出本地”的部署目标
- 安全边界可以显著收敛到单机与单网段

如果用对外沟通的业务语言，可以说这是“最接近百分百本地安全”的部署形态。

但如果用严格工程语言表述，更准确的说法是：

> 它不是数学意义上的“100% 安全”，而是当前架构下最强的数据本地化与最小外部暴露方案。

## Why A Full Local Option Matters

这条路线重要，是因为并不是所有 `Mira Light` 场景都需要云端智能。

很多现实部署里，更重要的其实是：

- 观众数据不要离场
- 图像与语音不要上云
- 设备控制链不要依赖公网
- 断网后仍然可以继续工作
- 安全审计边界尽量简单

对于展厅、企业内网、实验室、私密空间或含敏感人流数据的环境，这类要求往往比“模型能力再强一点”更优先。

## Architectural Principle

完整本地部署的推荐形态是：

```text
用户
-> 本地 OpenClaw / 本地模型
-> 本地 plugin
-> 本地 bridge
-> 本地 runtime / vision / audio
-> 本地热枢纽
-> ESP32 Mira Light
```

必要时还可以进一步收紧成：

```text
同一台本地主机
-> OpenClaw
-> bridge
-> runtime
-> vision
-> audio
-> 内网 ESP32
```

在这条形态下，整个系统的外部依赖可以被压缩到最低。

## What The Repository Already Supports

当前仓库已经具备支撑完整本地部署的多项基础能力：

- 本地 `OpenClaw` 节点模板与 rollout 方案
- 本地 `mira-light-bridge` plugin
- 本地 `bridge_server.py`
- 本地 `runtime`
- 本地 `vision` 栈
- 本地 `audio / tts / speak` 路径
- 本地 workspace identity 与 memory index

这意味着当前仓库不是只能走“云端大脑”。

相反，它已经具备了把 `Mira` 做成一个本地常驻 embodied agent 的大部分骨架。

## Why Full Local Can Be More Secure

完整本地部署之所以安全边界更强，不是因为“本地天然神奇”，而是因为它减少了几个关键暴露面：

- 不需要把图像、语音、状态上传到云端模型
- 不需要公网往返的控制链
- 不需要远端 bridge / 隧道 / 跨公网转发
- 不需要把设备控制面暴露给远端系统

这会直接带来几个结果：

- 数据更容易保证停留在本地磁盘和本地内存
- 审计范围更小
- 网络攻击面更窄
- 断网时系统仍可能继续运行
- 现场行为更可预测

## What “Data Never Leaves Local” Means In Practice

在完整本地部署里，“数据不出本地”应理解为以下目标同时成立：

- 模型推理使用本地模型或本地模型服务
- 记忆索引与检索在本地完成
- 图像帧不上传外部 API
- 语音转文本与文本转语音尽量使用本地能力
- bridge、runtime、vision、audio 都只运行在本地网络平面
- ESP32 只与本地节点通信

如果这些条件都满足，那么系统中的核心数据流就可以基本收敛在：

- 本地主机
- 本地局域网
- 本地可控存储

## Where The “100% Security” Claim Needs Care

这里必须诚实说明一件事：

技术上不应该轻易承诺“100% 安全”。

因为即使系统完全本地，也仍然可能存在：

- 本机操作系统漏洞
- 局域网内部风险
- 物理接触风险
- 运维误配置
- 插件或脚本自身 bug

所以最稳妥的正式表述不是“绝对百分百安全”，而是：

- 数据不出本地
- 外部依赖最小化
- 暴露面最小化
- 在当前架构中实现最强本地安全边界

如果是面向非技术受众，也可以用更容易理解的话说：

> 在合适条件下，`Mira Light` 可以做成完整本地闭环，核心数据无需离开本地网络，安全性也会比云端链路方案更高、更可控。

## What Full Local Is Good At

完整本地部署特别适合以下场景：

- 展览现场要求断网也能运行
- 图像、语音、交互数据不能离场
- 企业或机构要求数据主权
- 不希望依赖第三方云 API
- 需要更快的本地身体响应
- 需要更清晰的安全审计边界

## What Full Local Gives Up

这条路线也不是没有代价。

相对云端方案，完整本地部署通常会牺牲：

- 最强模型能力的即插即用性
- 云端长记忆与远程调度的便利性
- 跨地点统一运维的灵活性
- 某些云端 provider 已有能力的直接复用

因此它更像是一种：

- 用更强的数据主权
- 换取一定模型弹性与远程便利

的选择。

## Deployment Recommendation

如果目标是“本地闭环、数据不出本地、尽量减少外部依赖”，当前最推荐的部署分工是：

### 本地主机负责

- `OpenClaw` 或本地模型推理
- workspace 与 memory
- plugin
- bridge
- runtime
- vision
- audio / speak

### 本地热枢纽负责

- 局域网接入
- 与 ESP32 的稳定通信
- 必要时承担守护、日志与恢复

### ESP32 负责

- 动作执行
- 状态暴露
- 保持设备控制 API 简洁

## Practical Conclusion

完整本地部署不是退而求其次的备选项。

在很多要求高隐私、高可控、高可靠的场景里，它反而是更优先的主方案。

更准确地说：

> 如果你的首要目标是数据不出本地、断网可运行、以及最小化外部攻击面，那么 `Mira Light` 完整本地部署是完全合理且值得优先推进的。

## Recommended External Framing

> `Mira Light` 在合适条件下可以实现完整本地部署：模型、记忆、bridge、runtime、vision、audio 与设备执行全部留在本地网络中完成。
> 这意味着核心交互数据无需离开本地环境，系统也不必依赖公网链路运行。
> 从工程实践上看，这是一种最接近“数据不出本地、最强本地安全边界”的部署方案。
