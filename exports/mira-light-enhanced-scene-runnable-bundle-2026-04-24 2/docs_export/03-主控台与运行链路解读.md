# 主控台与运行链路解读

这份文档解释这个增强包里的主控台、bridge、runtime、scene 是怎么串起来的。

## 默认运行链

当前增强包默认仍然采用 release 版稳定链路：

```text
浏览器
-> 导演台 console
-> 本地 bridge
-> runtime
-> 真实台灯 / dry-run / mock
```

关键位置：

- 终端运行入口：`scripts/booth_controller.py`
- runtime 启动 scene：`scripts/mira_light_runtime.py`
- runtime 事件触发：`scripts/mira_light_runtime.py`
- 导演台：`scripts/console_server.py`
- bridge API：`tools/mira_light_bridge/bridge_server.py`

## 为什么这个增强包还额外放了 motion scripts

因为主仓已经把每个主场景拆成了：

```text
Motions/<scene>/scene_script.py
```

这一层的价值不是重复 choreography，而是：

- 给导演台更稳定的 per-scene launch manifest
- 给后续统一信号交付格式留出入口
- 把每个场景的 step outline 显式化

## 这个增强包怎么处理主控台差异

### release 版主控台

- 文件：`scripts/console_server.py`
- 优点：更稳定，适合新电脑先跑起来

### 主仓增强版主控台

- 文件：`scripts/console_server_with_motion_scripts.py`
- 配套：
  - `scripts/mira_light_signal_delivery.py`
  - `config/mira_light_signal_delivery.schema.json`
  - `Motions/`

它更接近主仓现在的组织方式，但我没有直接覆盖 release 默认入口，而是把它作为增强选项单独放出来。

## `track_target` 为什么最特殊

因为它不是单纯 scene。

它有两条路径：

### 路径 A：fallback choreography

- `scripts/scenes.py`

这版适合排练和没有视觉输入时的说明性演示。

### 路径 B：实时 tracking

- `scripts/vision_runtime_bridge.py`
- `scripts/mira_light_runtime.py:apply_tracking_event(...)`

这版才是“它真的在看见并跟随”的主路径。

## `farewell` 和 `touch_affection` 为什么也不是纯静态

- `farewell`
  动态 builder 会根据离场方向生成 scene
- `touch_affection`
  动态 builder 会根据来手方向调整 scene

这也是为什么在对照表里，这两个场景都标成了 `dynamic_builder`。
